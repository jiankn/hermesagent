# 扩展模式

## 容量阈值

| 规模 | 条目数 | 策略 |
|------|--------|------|
| 小 | <100 | 单个 memory.md，无命名空间 |
| 中 | 100-500 | 拆分为 domains/，基本索引 |
| 大 | 500-2000 | 完整命名空间层次结构，积极压缩 |
| 巨大 | >2000 | 按年归档，仅摘要热层 |

## 何时拆分

在以下情况下创建新的命名空间文件：
- 单个文件超过 200 行
- 主题有 10+ 个不同的纠正
- 用户明确分离背景（"对于工作..."，"在这个项目中..."）

## 压缩规则

### 合并类似的纠正
```
之前（3 个条目）：
- [02-01] 使用制表符不是空格
- [02-03] 用制表符缩进
- [02-05] 请用制表符缩进

之后（1 个条目）：
- 缩进：制表符（已确认 3 次，02-01 至 02-05）
```

### 摘要冗长的模式
```
之前：
- 给 Marcus 写电子邮件时，使用项目符号，保持在 5 项以下，
  不要行话，要点优先，他更喜欢早上发送

之后：
- Marcus 电子邮件：项目符号 ≤5，无行话，BLUF，偏好上午
```

### 带背景归档
移动到冷时：
```
## 已归档 2026-02

### 项目：old-app（自 2025-08 以来不活跃）
- 使用 Vue 2 模式
- 喜欢 Vuex 而不是 Pinia
- Jenkins 上的 CI（已弃用）

原因：项目已完成，模式不太可能适用
```

## 索引维护

`index.md` 跟踪所有命名空间：
```markdown
# 记忆索引

## 热（总是加载）
- memory.md: 87 行，更新 2026-02-15

## 温（匹配时加载）
- projects/current-app.md: 45 行
- projects/side-project.md: 23 行
- domains/code.md: 112 行
- domains/writing.md: 34 行

## 冷（归档）
- archive/2025.md: 234 行
- archive/2024.md: 189 行

上次压缩：2026-02-01
下次安排：2026-03-01
```

## 多项目模式

### 继承链
```
全局（memory.md）
  └── 领域（domains/code.md）
       └── 项目（projects/app.md）
```

### 覆盖语法
在项目文件中：
```markdown
## 覆盖
- 缩进：空格（覆盖全局制表符）
- 原因：项目 eslint 配置需要空格
```

### 冲突检测
加载时，检查冲突：
1. 构建继承链
2. 检测矛盾
3. 最具体的获胜
4. 记录冲突以供以后审查

## 用户类型适应

| 用户类型 | 记忆策略 |
|---------|---------|
| 高级用户 | 积极学习，最少确认 |
| 休闲用户 | 保守学习，频繁确认 |
| 团队共享 | 每用户命名空间，共享项目空间 |
| 注重隐私 | 仅本地，每个分类明确同意 |

## 恢复模式

### 背景丢失
如果代理在会话中途失去背景：
1. 重新读取 memory.md
2. 检查 index.md 以获取相关命名空间
3. 加载活跃项目命名空间
4. 继续使用恢复的模式

### 损坏恢复
如果记忆文件损坏：
1. 检查 archive/ 以获取最近的备份
2. 从 corrections.md 重建
3. 要求用户重新确认关键偏好
4. 记录事件以供调试
